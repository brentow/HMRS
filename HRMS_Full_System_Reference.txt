### HIGH-LEVEL APP FLOW

1. `LoginPage.xaml` starts the app.
2. `LoginViewModel` submits credentials to `LoginDataService`.
3. `LoginDataService` reads `user_accounts`, joins `employees` and `roles`, validates password hash, updates `last_login_at`, and auto-creates missing employee user accounts when needed.
4. Successful login opens `DashboardWindow` with an `AuthenticatedUser` object.
5. `DashboardWindow` applies role-based navigation and hosts the major module views.
6. `DashboardViewModel` loads dashboard snapshot data through `DashboardDataService`.
7. `SystemRefreshBus` is used to notify the dashboard and visible modules when shared data changes.

### ROLES, ACCESS, AND AUTHENTICATION

#### Seeded roles in migrations

- Admin
- HR Manager
- Dept Head
- Employee

#### Runtime-created role

- Beneficiary
  - created on demand by `BeneficiaryDataService` if the role does not exist yet

#### Login/account behavior

- Password hashes use PBKDF2 SHA-256 via `PasswordSecurity`
- Locked and inactive accounts are blocked by `LoginDataService`
- Beneficiary accounts approved from staging are created as active accounts and forced to change password on first login
- Reset passwords for other users also force a change on next login
- Employee accounts can be auto-generated if an employee record exists without a linked `user_accounts` row

#### Access control helpers

- `AuthorizationGuard.GetRoleNameAsync`
- `AuthorizationGuard.IsAdmin`
- `AuthorizationGuard.IsAdminOrHr`
- `AuthorizationGuard.DemandAdmin`
- `AuthorizationGuard.DemandAdminOrHr`

#### Dashboard role behavior

- Admin: full module access
- HR Manager: broad HR access, including beneficiary verification
- Employee: dashboard, attendance, adjustments, leave, payroll, documents, development, profile
- Guest/default fallback: dashboard + profile

### PRACTICAL HANDOFF NOTES FOR FUTURE WORK

- Do not treat `MainWindow.xaml` as the real shell; the real startup window is `View/LoginPage.xaml`.
- The actual source code is inside the nested `HRMS/` folder.
- Ignore `_build_out/` copies when tracing behavior; they are output artifacts.
- Beneficiary import and GGMS sync are already implemented in source, not just planned.
- Payroll currently has hardcoded GGMS office mapping for HRMS:
  - Office ID `18`
  - Office Code `OFF-2026-0007`
- Documents and notifications depend on the logged-in user being linked to a valid employee profile.
- Beneficiary accounts are not employee-linked; they are created through `BeneficiaryStaging -> user_accounts`.
- User/profile data, attendance, leave, payroll, training, recruitment, and beneficiary verification all raise or depend on shared refresh patterns.
- If working on database changes, check the SQL migration chain first and keep `schema_migrations` in sync.
